## 1. 프로젝트 개요

기존에는 부동산 매물 정보를 엑셀로 관리하고 있어, 위치와 매물 정보를 한눈에 파악하기 어렵다는 불편이 있었습니다.  
이를 개선하기 위해 지도 기반으로 매물 정보를 조회하고 관리할 수 있는 프로그램을 개발했습니다.

### 개발 목적
- 엑셀 중심의 매물 관리 방식 개선
- 지도 기반으로 매물 위치와 정보를 직관적으로 확인
- 매물 등록, 수정, 삭제를 포함한 관리 기능 제공

---

## 2. 주요 기능

### 1) 지도 기반 매물 조회
- 네이버 지도 API를 활용해 지도 위에서 매물 위치와 정보를 확인할 수 있습니다.
- 지도 기반으로 매물을 직관적으로 조회할 수 있도록 구현했습니다.

### 2) 매물 등록·수정·삭제
- 부동산 매물 정보를 등록, 수정, 삭제할 수 있습니다.
- 매물 데이터 관리 기능을 통해 기본적인 CRUD를 구현했습니다.

### 3) 사진 업로드 및 관리
- 매물 사진을 업로드하고 관리할 수 있습니다.
- 사진 파일과 매물 정보를 함께 저장·조회할 수 있도록 구성했습니다.

### 4) 메모 및 옵션 관리
- 매물별 메모와 옵션 정보를 함께 저장하고 확인할 수 있습니다.
- 사용자가 매물 상태를 더 구체적으로 기록할 수 있도록 구성했습니다.

### 5) 조건 기반 매물 필터링
- 거래 유형, 상태, 옵션 등의 조건을 기반으로 원하는 매물을 구분할 수 있습니다.
- 필요한 매물을 효율적으로 확인할 수 있도록 구현했습니다.

## 4. 아키텍처

<img width="655" height="808" alt="architecture" src="https://github.com/user-attachments/assets/e52298d7-6386-4ed9-b424-0dca33755004" />

사용자 요청은 브라우저와 Thymeleaf 기반 화면을 통해 Spring Boot 서버로 전달되며,  
서버는 Controller-Service-Repository 구조로 비즈니스 로직과 데이터 처리를 담당합니다.  
매물 데이터는 실사용 환경에서는 SQLite에 저장하고, 배포 테스트 환경에서는 PostgreSQL/AWS RDS를 활용했습니다.  
또한 지도 기능은 네이버 지도 API 및 Geocoding API와 연동하고, 매물 사진은 파일 저장소와 DB 메타데이터를 분리해 관리하도록 구성했습니다.

## 7. 트러블슈팅

1) 개발·배포 환경 분리

소규모 사용자 대상 프로그램이라는 점을 고려해,
실사용 환경은 SQLite 기반 로컬 구조로 구성하고 배포 테스트 환경은 PostgreSQL/AWS RDS로 분리했습니다.

2) 네이버 지도 API 연동 오류 해결

구버전 인증 파라미터 사용으로 지도 기능이 정상 동작하지 않는 문제가 있었습니다.
공식 문서를 바탕으로 최신 방식으로 수정해 지도 기능을 정상화했습니다.

3) 등록 폼 입력값 초기화 문제 해결

등록 폼에서 검증 오류가 발생하면 입력값이 모두 초기화되는 문제가 있었습니다.
검증 실패 이후에도 기존 입력값이 유지되도록 처리해 사용자 재입력 부담을 줄였습니다

## 10. 회고

처음 친구에게 의뢰를 받았을 때, 누군가가 실제로 사용하는 프로그램을 만든다는 점이 무척 흥미롭게 느껴졌습니다. 단순히 기능을 구현하는 것을 넘어, 실제 사용자를 위한 제품을 만든다는 생각에 설렘도 컸습니다.
하지만 실제로 사용되는 프로그램을 만들기 위해서는 생각보다 훨씬 많은 요소를 고려해야 했습니다. 
이전에는 “이 정도면 괜찮겠다”고 생각했던 부분도, 실제 사용자 입장에서는 불편이 될 수 있다는 점을 알게 되었습니다. 대표적으로 등록 과정에서 잘못 입력한 값이 있을 때 기존 입력값이 모두 초기화되는 문제를 개선하려고 했던 경험이 기억에 남습니다.
이 프로젝트를 통해 하나의 제품을 만든다는 것은 단순한 기능 구현이 아니라, 사용자 경험과 디테일을 끝까지 고민하는 과정이라는 점을 배울 수 있었습니다.
